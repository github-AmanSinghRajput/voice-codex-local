# v1.1 Multi-Coding-Assistant Provider Plan

## Goal

Expand the app beyond Codex-only so users can connect either OpenAI Codex or Anthropic Claude Code — using their existing subscriptions with zero extra API cost.

## Core Principle

Users already pay $20/month for ChatGPT Plus (Codex) or Claude Pro/Max (Claude Code). The app wraps these CLI tools directly, so there is no API key recharge, no wallet top-up, no per-token billing. The user's subscription covers everything.

## Why CLI Wrapping, Not Direct API

| Approach | Cost to user | Why |
|---|---|---|
| Wrap CLI (`codex exec`, `claude -p`) | $0 extra — uses existing subscription | User already pays $20/month. No reason to charge twice. |
| Direct API (`api.anthropic.com`, `api.openai.com`) | Extra — requires separate API credits | Bad product experience. Subscription already covers the capability. |

CLI wrapping is the right architecture for a free desktop product where users bring their own subscription.

## Provider Comparison

| Capability | Codex CLI | Claude Code CLI |
|---|---|---|
| Single-shot prompt | `codex exec --prompt "..."` | `claude -p "..." --bare` |
| Check login status | `codex login status` | `claude auth status` |
| Structured JSON output | `--output-schema '{...}'` | `--output-format json --json-schema '{...}'` |
| Login flow | `codex login --device-auth` → browser OAuth | `claude` → browser OAuth |
| Auth credential storage | Local file | macOS Keychain (macOS), `~/.claude/.credentials.json` (Linux) |
| Read-only / advisory mode | `--sandbox read-only` | `--allowedTools "Read"` |
| Write-access mode | `--sandbox workspace-write` | `--allowedTools "Read,Edit,Bash"` |
| Skip slow startup | N/A | `--bare` (skips hooks, MCP, plugins) |
| Streaming output | File-based output | `--output-format stream-json` (newline-delimited JSON) |
| Continue conversation | N/A | `--continue` flag |

The mapping is nearly 1:1. Both CLIs support single-shot prompts with structured JSON output, auth status checks, and sandboxed execution modes.

## Backend Architecture

### Current state (v1.0)

`codex-client.ts` directly shells out to `codex exec` for all operations:
- `generateAssistantReply()` → read-only prompt
- `decideWriteIntent()` → read-only prompt with output schema
- `executeApprovedWrite()` → workspace-write prompt
- `getCodexStatus()` → login status check
- `collectGitDiff()` → git diff after write

### Target state (v1.1)

Replace `codex-client.ts` with a provider abstraction:

```
interface CodingAssistantProvider {
  id: string;                          // 'codex' | 'claude'
  name: string;                        // 'OpenAI Codex' | 'Anthropic Claude Code'
  checkAuth(): Promise<AuthStatus>;
  generateReply(prompt, history, workspace): Promise<ReplyResult>;
  decideWriteIntent(prompt, history, workspace): Promise<WriteDecision>;
  executeApprovedWrite(approval, history, workspace): Promise<ExecutionResult>;
}
```

Two implementations:

```
CodexProvider implements CodingAssistantProvider:
  checkAuth()          → run `codex login status`, parse output
  generateReply()      → run `codex exec --sandbox read-only --prompt "..."`
  decideWriteIntent()  → run `codex exec --sandbox read-only --output-schema '{...}' --prompt "..."`
  executeApprovedWrite() → run `codex exec --sandbox workspace-write --prompt "..."`

ClaudeProvider implements CodingAssistantProvider:
  checkAuth()          → run `claude auth status`, parse JSON (exit code 0 = logged in)
  generateReply()      → run `claude -p "..." --bare --allowedTools "Read" --output-format json`
  decideWriteIntent()  → run `claude -p "..." --bare --allowedTools "Read" --output-format json --json-schema '{...}'`
  executeApprovedWrite() → run `claude -p "..." --bare --allowedTools "Read,Edit,Bash" --output-format json`
```

### Provider selection

- Configured via env: `ASSISTANT_PROVIDER=codex` or `ASSISTANT_PROVIDER=claude`
- Selectable in the UI: provider picker in onboarding or settings
- The app detects which CLIs are installed and shows available options
- Only one provider active at a time per session (simplest for v1.1)

## Frontend Changes

### Onboarding

Currently the onboarding screen only guides Codex login. For v1.1:

1. Show detected providers (which CLIs are installed)
2. Let user choose: "Connect with Codex" or "Connect with Claude Code"
3. Guide the appropriate login command
4. Detect auth state for the selected provider

### UI copy

All references to "Codex" in the UI become dynamic:
- "Codex is listening" → "{provider} is listening"
- "Sending to Codex" → "Sending to {provider}"
- "Codex replied" → "{provider} replied"

### Settings

Add provider selection in the settings drawer:
- Show current provider
- Allow switching (requires the other CLI to be installed and authenticated)
- Show auth status for each detected provider

## What Stays The Same

The following are provider-agnostic and require no changes:

- Approval flow (propose → review diff → approve/reject)
- Voice capture (getUserMedia → STT → transcript text)
- TTS playback (Kokoro / browser fallback)
- Diff review UI
- Text chat with markdown rendering
- Integrated shell terminal
- Workspace/project boundary selection
- Secret policy enforcement
- Conversation persistence in Postgres

The entire product experience is identical regardless of which provider the user connects. The only difference is which CLI processes the prompts.

## Auth Status Detection

### Codex

```bash
codex login status
# Output: text describing login state
# Parse for "logged in" indicator
```

### Claude Code

```bash
claude auth status
# Output: JSON with auth type, account info
# Exit code: 0 = logged in, 1 = not logged in

claude auth status --text
# Human-readable output
```

Both can be run as child processes and parsed from the backend.

## Installation Detection

Before showing a provider option, check if the CLI exists:

```bash
which codex    # or: codex --version
which claude   # or: claude --version
```

If neither is installed, the onboarding screen should guide the user to install one.

## Implementation Order

1. Create `CodingAssistantProvider` interface
2. Extract current Codex logic into `CodexProvider`
3. Implement `ClaudeProvider` following the same pattern
4. Add provider detection (which CLIs are installed)
5. Add provider selection in onboarding UI
6. Update all UI copy to use dynamic provider name
7. Add provider switching in settings
8. Test both providers end-to-end with the full voice + approval flow

## Risks

- **Claude Code CLI may change flags/output format** — pin to a minimum version and document it
- **Auth token expiry** — both CLIs handle refresh internally, but the app should detect and surface re-login prompts gracefully
- **Different response quality/style** — the system prompt and write-decision schema may need per-provider tuning
- **Claude Code `--bare` mode may skip context** — test that workspace-aware prompts still work correctly in bare mode

## Non-Goals for v1.1

- Supporting more than two providers
- Running multiple providers simultaneously in one session
- API-key-based direct integration (reserved for future if needed)
- Enterprise/team account management
